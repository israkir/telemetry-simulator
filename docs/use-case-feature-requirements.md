# Use-Case and Feature Requirements

This document maps **observability use cases** and **feature requirements** to scenario definitions and telemetry: which conditions the simulator covers, what each scenario does, and how well generated data answers specific questions.

**Attribute names:** Examples use the **`gentoro.*`** prefix from the bundled semantic convention. When using another vendor prefix (`VENDOR` or `--vendor`), use that prefix in queries (e.g. `vendor.tenant.id` instead of `gentoro.tenant.id`).

---

## 1. Tool calls have higher latency

**When:**

- During peak hours (weekdays 9 am–2 pm PT, also post long weekends / holidays)
- When claim status output is “technician on-route to scheduled appointment”
- When scheduling appointments in certain zip codes


| Condition                                       | What the scenario does                                                                                                                                                                                | Scenarios                                                                                                                                |
| ----------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| Peak hours / post long weekend                  | Traces use a higher-latency profile so tool and orchestration spans take longer. The condition (`peak_hours` or `post_long_weekend`) is recorded on span attributes so you can filter or slice by it. | `generic_higher_latency_peak_hours`, `new_claim_electronics_higher_latency_peak_hours`, `new_claim_appliances_higher_latency_peak_hours` |
| Claim status = “technician on-route”            | Same claim_status flow with higher latency; conversation samples include both a generic outcome and a technician-on-route outcome. The condition is on span attributes.                               | `claim_status_phone_higher_latency`                                                                                                      |
| Scheduling in certain zip codes                 | `update_appointment` runs with higher latency; the scenario records the zip (e.g. 90210 for phone, 10001 for electronics) on span attributes so you can correlate latency with region.                | `update_appointment_phone_higher_latency`, `update_appointment_electronics_higher_latency`                                               |
| Cancel claim with appointment already scheduled | `cancel_claim` runs with higher latency; the scenario records the condition (e.g. `appointment_scheduled`) on span attributes.                                                                        | `cancel_claim_appliances_higher_latency`                                                                                                 |

## 2. Agent calls correct tools with incorrect parameters (4xx errors)

**Cause:** How the input entities were specified (e.g. different ways dates are specified by user, claim ID entered with or without dash). Wrong format or missing parameters result in 4xx errors.


| Condition                             | Intention                                                                                                                                                                                                                                                                           | Product context                                                           | Scenarios                                                                                                                                           |
| ------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------- |
| Date format or missing date           | Agent calls `new_claim` with a bad or missing date (e.g. “13/31/2024”, “March 15”, or omitted). One MCP attempt is marked failed with `invalid_arguments` and an exception message (e.g. invalid date or missing required field). Trace shows the wrong arguments on the tool span. | Validation errors on claim creation (date format, required fields).       | `new_claim_appliances_tool_4xx_invalid_params`, `new_claim_electronics_tool_4xx_invalid_params`                                                     |
| Claim ID without dash or wrong prefix | Agent calls `claim_status` with a claim ID like “8842”, “7733”, or “8821” instead of “PH-8842”, “HA-7733”, or “EL-8821”. Same pattern for all divisions. One MCP attempt fails with `invalid_arguments` and an exception about claim ID format.                                                                                          | User gave only the number or wrong prefix for status checks.              | `claim_status_phone_tool_4xx_invalid_params`, `claim_status_appliances_tool_4xx_invalid_params`, `claim_status_electronics_tool_4xx_invalid_params`   |
| Invalid or missing cancel reason      | Agent calls `cancel_claim` with an empty or free-text reason instead of an allowed value (e.g. customer_requested, duplicate, no_longer_needed). One MCP attempt fails with `invalid_arguments`. Same pattern for all divisions.                                                    | Validation of cancel reason (different 4xx context than claim ID format). | `cancel_claim_phone_tool_4xx_invalid_params`, `cancel_claim_appliances_tool_4xx_invalid_params`, `cancel_claim_electronics_tool_4xx_invalid_params` |
| Invalid or missing policy_id          | Agent calls `cancel_product` with an empty or malformed policy_id (e.g. "123" instead of "pol_xxx"). One MCP attempt fails with `invalid_arguments`. Same pattern for all divisions.                                                                                                  | Validation of policy ID for product/policy cancellation.                 | `cancel_product_phone_tool_4xx_invalid_params`, `cancel_product_appliances_tool_4xx_invalid_params`, `cancel_product_electronics_tool_4xx_invalid_params` |

## 3. Agent calls incorrect tools

**Cause:** Same input cluster but maps to tools with same functionality but from different divisions. Reasons:

- Tool descriptions are similar (cancel product vs cancel claim; tools from different division with similar functionality)
- User requests are ambiguous


| Condition                                           | Intention                                                                                                                                                                                                                                                                         | Product context                                                                                | Scenarios                                                       |
| --------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------- | --------------------------------------------------------------- |
| Similar tool names (cancel product vs cancel claim) | User says “cancel my claim”; the agent calls `cancel_product` instead of `cancel_claim`. The tool succeeds but the response-compose step is marked failed with WrongToolError so wrong-tool traffic shows up in error metrics. The trace shows the tool that was actually called. | Same user intent → similar tools (cancel_claim vs cancel_product); agent picks the wrong tool. | `agent_confusion_cancel_product_instead_of_claim`               |
| Ambiguous request → wrong division (pay)            | User wants to pay for a phone claim; the request is ambiguous. Agent routes to the electronics MCP; trace shows electronics pay with wrong division. MCP call fails with `step.outcome=fail`, `error.type=tool_error`, and optional exception (e.g. WrongDivisionError).          | Same input cluster (pay) → same tool name across divisions; agent picks wrong division.        | `agent_confusion_pay_phone_wrong_division_electronics`          |
| Ambiguous request → wrong division (cancel claim)   | User wants to cancel a phone claim; the request is ambiguous. Agent routes to the electronics MCP; trace shows electronics cancel_claim with wrong division. MCP call fails with `step.outcome=fail`, `error.type=tool_error`, and optional exception.                            | Similar tool names and wording across divisions; agent picks wrong division.                   | `agent_confusion_cancel_claim_phone_wrong_division_electronics` |


Same input cluster, different division: the two wrong-division scenarios above both represent “user intent maps to pay or cancel claim but the agent uses the other division’s MCP”; the trace shows the wrong `mcp.server.uuid` / `mcp.tool.uuid`.

## 4. Agent calls partial set of tools in a workflow

**Cause:** No context about sequence.

- Doesn’t call the full set of tools in new-claim workflow
- First tool called is not the first tool in expected sequence


| Condition                                 | Intention                                                                                                                                                                                                             | Product context                                                                   | Scenarios                                    |
| ----------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------- | -------------------------------------------- |
| Partial tool set (new_claim never called) | User wants to start a claim. The trace shows planner, task, tool recommendation, then response compose—no MCP tool execution for `new_claim`. The response-compose step is marked failed (e.g. PartialWorkflowError). | Partial set of tools in the new-claim workflow; agent does not call the full set. | `agent_confusion_new_claim_partial_tools`    |
| Wrong tool order                          | User wants to start a claim. The trace shows the agent calling `update_appointment` first, then `new_claim`. Response compose is marked failed (e.g. WrongToolOrderError).                                            | First tool called is not the first in the expected sequence.                      | `agent_confusion_new_claim_wrong_tool_order` |

## 5. Agent output contains sentences that don’t originate from tool output

**Cause:** Retrieval of product info summarized incorrectly (ungrounded / hallucinated output).


| Condition               | Intention                                                                                                                                                                                                                                                                                                                                        | Product context                                             | Scenarios                                               |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------- | ------------------------------------------------------- |
| Incorrect summarization | The agent correctly calls `claim_status` and the tool succeeds. The response-compose step is marked failed with UngroundedContentError: the agent’s reply contains sentences that don’t originate from the tool (e.g. tool says in_review but agent says “approved” or invents appointment times). Conversation samples illustrate the mismatch. | Correct tool and MCP; response not grounded in tool output. | `agent_confusion_claim_status_ungrounded_summarization` |
